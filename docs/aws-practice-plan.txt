==============================================================================
PHASE A — ACCOUNT + COST SAFETY (do this BEFORE anything else)
==============================================================================
  A1. Create the AWS account (or use existing). Pick ONE region, close to you, and
      stick to it (mixing regions = confusion + egress charges).
  A2. STOP using the root user for daily work. Create an IAM admin user + enable MFA
      on root. Log in as the IAM user from now on.
  A3. COST GUARDRAILS (the most important step):
      - Billing -> Budgets -> create a monthly budget of $1 with email alerts at 50/80/100%.
      - Enable "Receive Billing Alerts", then a CloudWatch billing alarm at $1.
      - Turn on Cost Explorer so you can see spend daily.
      This way a mistake pings you instantly instead of a $200 surprise.

==============================================================================
PHASE B — NETWORK BASICS (keep it simple, default VPC is fine)
==============================================================================
  B1. Use the default VPC (no need to build one for practice).
  B2. Create/pick a Security Group for EC2:
      - inbound 22 (SSH) from YOUR IP only (not 0.0.0.0/0).
      - inbound 8100 (or 80/443) only if you want to hit the dashboard; else leave closed.
  B3. Create a Security Group for RDS:
      - inbound 5432 FROM THE EC2 SECURITY GROUP ONLY (not public). RDS stays private.
  (Interview point: "the DB is not publicly reachable; only the app SG can hit 5432.")

==============================================================================
PHASE C — EC2 (deploy the app) -> "deployed a containerized app on EC2"
==============================================================================
  C1. Launch EC2 t2.micro: Ubuntu 22.04 (or Amazon Linux 2023), the EC2 SG, a new key pair.
  C2. SSH in with the key pair (chmod 400 the .pem).
  C3. Install Docker + the compose plugin:
      Ubuntu: apt update && apt install -y docker.io docker-compose-plugin && systemctl enable --now docker
  C4. ADD SWAP (1 GB RAM is tight for Playwright/Chromium card rendering):
      fallocate -l 2G /swapfile; chmod 600 /swapfile; mkswap /swapfile; swapon /swapfile;
      echo '/swapfile none swap sw 0 0' >> /etc/fstab
  C5. Get the code: git clone the publisher repo (GitHub). Create a .env with TEST/dummy
      keys (see Phase G). Point DATABASE_URL at RDS (after Phase D).
  C6. docker compose up -d --build. Confirm containers healthy. Optional: hit the dashboard.
  (Interview points: "why EC2 vs Fargate/ECS", "swap for a memory-tight instance",
   "docker compose for local-parity deploys".)

==============================================================================
PHASE D — RDS POSTGRES -> "used RDS (managed Postgres)"
==============================================================================
  D1. Create RDS Postgres db.t3.micro, 20 GB gp2, single-AZ, the RDS SG, NOT publicly
      accessible. Set a master user + password.
  D2. From EC2, set DATABASE_URL=postgresql://USER:PASS@<rds-endpoint>:5432/publisher
      in .env. Drop the local postgres service from the compose (or leave it unused).
  D3. Run the app's create_tables / migrations against RDS; confirm it connects.
  (Interview points: "RDS vs self-managed Postgres: backups, patching, failover managed
   for you", "single-AZ for cost vs multi-AZ for HA".)

==============================================================================
PHASE E — S3 -> "used S3 for object storage / backups"
==============================================================================
  E1. Create a private S3 bucket (block all public access ON).
  E2. IAM: create a policy + user/role with least-privilege access to that bucket only.
  E3. Point src/backup.py at S3 (it is already boto3 / S3-compatible — swap the endpoint
      + creds from B2 to AWS S3). Run a backup -> object appears in the bucket.
  (Interview points: "S3 for durable object storage + backups", "least-privilege IAM
   policy scoped to one bucket", "lifecycle rules to expire old backups".)

==============================================================================
PHASE F — (OPTIONAL) LAMBDA -> "wrote + deployed a serverless function"
==============================================================================
  F1. One small slice, e.g. a scheduled Lambda (EventBridge cron) that writes a heartbeat
      to S3, or triggers the publish check. Free forever.
  (Interview point: "when serverless vs a long-running container".)

==============================================================================
PHASE G — SECRETS SAFETY (do NOT leak prod)
==============================================================================
  G1. The AWS box uses TEST/dummy keys ONLY (fake LinkedIn/X/NVIDIA/Deepgram/OpenRouter).
      It does NOT need to post for real; run it in a safe/dry mode.
  G2. Never commit .env. Consider AWS SSM Parameter Store / Secrets Manager for the values
      (itself a nice résumé line: "stored config in SSM Parameter Store, not in the repo").

==============================================================================
PHASE H — DOCUMENT IT (this is what actually gets you the job)
==============================================================================
  H1. A one-page architecture note + a simple diagram (EC2 -> RDS -> S3).
  H2. INTERVIEW TALKING POINTS written out (the real deliverable):
      - EC2 vs Fargate/ECS; why docker compose.
      - RDS vs self-managed PG; single vs multi-AZ.
      - Security groups: app-only access to the DB, SSH from your IP.
      - S3 + least-privilege IAM; lifecycle rules.
      - Cost controls (budgets + billing alarm) — shows ops maturity.
      - What you would change for PRODUCTION (multi-AZ, ALB, autoscaling, ECS/Fargate,
        IaC with Terraform, CI/CD to deploy). Being able to say "here is the prod version"
        is what senior means.
  H3. Add it to the résumé + a short LinkedIn "building in public" post (real experience).

==============================================================================
PHASE I — TEARDOWN / FREE-TIER DISCIPLINE
==============================================================================
  I1. STOP the EC2 + RDS when not actively practicing (stopped instances do not burn hours,
      though RDS storage still counts — delete RDS if done for a while, keep a final snapshot).
  I2. Empty + delete the S3 bucket when done. Confirm $0 in Cost Explorer.
  I3. Screenshot the console (proof/portfolio) before teardown.
