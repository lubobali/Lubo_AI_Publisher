GOAL: Grow Lubo Bali's social media audience (LinkedIn first, then X,
Instagram, Substack) by posting daily, high-quality AI/Data Engineering
content with real screenshots. AI-generated posts go through approval
queue. Manual posts also supported — everything tracked in one system.

==============================================================================
1. WHAT IT IS
==============================================================================

A standalone Python service that runs on Hetzner, completely separate from
LuBot. Every day it:

  1. Wakes up at a random time (weekdays 4-10pm CT, weekends 10am-6pm CT)
  2. Picks today's topic from a rotating calendar
  3. Scrapes the web for the latest, most interesting content on that topic
  4. AI (NVIDIA 253B) writes a LinkedIn post in Lubo's voice
  5. Playwright takes 1-2 real screenshots (websites, terminals, dashboards)
  6. Saves post to approval queue (status: PENDING)
  7. Lubo approves/rejects from web dashboard
  8. Approved posts publish to LinkedIn (and later X, Instagram, Substack)
  9. Tracks engagement metrics and feeds them back into future posts

Lubo can also create manual posts from the dashboard — same tracking,
same analytics, same system. Everything in one place.

No third-party posting tools (no Buffer, no Typefully). Direct platform APIs.
Inspired by Kevin Naughton's Ferryman ($10k MRR SaaS built on same API).

4. POSTING FLOW (how a post gets created)
==============================================================================

  Step 1 — SCHEDULE (scheduler.py)
    - Night before: pre-calculate tomorrow's random post time
    - Weekdays: random between 4:00 PM - 10:00 PM CT
    - Weekends: random between 10:00 AM - 6:00 PM CT
    - Never same time two days in a row (min 1hr difference)
    - Cron wakes up at calculated time

  Step 2 — TOPIC SELECTION (topic_rotator.py)
    - Check rotating calendar for today's topic category
    - Topics rotate weekly so Monday is NOT always the same topic
    - Week 1: Mon=AI News, Tue=Tech Talk, Wed=AI Gadgets...
    - Week 2: Mon=Biohacker, Tue=Big Tech, Wed=DE Work...
    - Shifts by 1 position each week

  Step 3 — SCRAPE (scraper.py)
    - Search multiple sources for today's topic (not just first Google result)
    - Find 3-5 candidate stories
    - Rank by recency, relevance, and uniqueness
    - Check against duplicate_checker.py (URL + embedding similarity)
    - Pick the best non-duplicate story
    - Sources per category configured in scraper_sources.yaml

  Step 4 — WRITE (writer.py)
    - Send to NVIDIA 253B with:
      a. Today's topic + scraped content
      b. Voice rules (casual ESL tone, no AI slop)
      c. 3-5 sample posts for style matching
      d. Analytics feedback (what topics performed best recently)
      e. Instruction to suggest screenshot target URL
    - Output: post text + screenshot URL

  Step 5 — SCREENSHOT (screenshotter.py)
    - Playwright navigates to the suggested URL
    - Takes screenshot at 1200x627px (LinkedIn optimal)
    - Dark mode when possible
    - Crops browser chrome — looks like a natural screenshot
    - For terminal screenshots: use actual terminal output captures
    - Save to local storage with timestamp

  Step 6 — POST (linkedin_client.py)
    - Initialize image upload via /rest/images
    - Upload screenshot binary
    - Create post via /rest/posts with:
      author: urn:li:person:{LUBO_ID}
      commentary: the AI-written post
      content.media.id: the uploaded image URN
      visibility: PUBLIC
      distribution.feedDistribution: MAIN_FEED
    - Save LinkedIn post URN to database

  Step 7 — LOG (db.py)
    - Save everything to publisher_posts table
    - Topic, text, image, source URL, post URN, timestamp

6. WRITING STYLE RULES (voice_rules.yaml)
==============================================================================

  CORE VOICE:
    - Casual, confident, real person talking
    - ESL natural English — not perfect grammar, but always makes sense
    - Like texting a friend who happens to be a senior engineer
    - Lubo's actual personality: direct, technical, no BS

  STRUCTURE:
    - First 2 lines MUST be a catchy hook (this is what people see before
      "see more" — most important part of the post)
    - Short paragraphs, 1-3 sentences each
    - Total length: 400-1500 characters (varies — mix short and long posts)
    - End with a question OR hot take to drive comments

  DO:
    - Use specific numbers ("50GB file, 3.2 seconds", "120K lines of code")
    - Include personal experience ("I tested this", "we built this")
    - Be opinionated ("everyone hyping X but honestly Y is better because")
    - Break lines often, white space is your friend
    - Reference real tools by name (not "a popular framework")

  DO NOT:
    - No dashes (—) or en-dashes
    - No special characters humans dont normally type
    - No bullet points with fancy symbols
    - No "In todays rapidly evolving landscape" or any AI slop phrases
    - No perfect grammar — occasional typos OK, missing articles OK
    - No emojis except maybe 1-2 if natural (fire, rocket, thats it)
    - No "Lets dive in" or "Heres the thing" or "Game changer"
    - No external links in post body (kills LinkedIn reach)
    - No walls of text — if its more than 3 sentences, break it up
    - Never sound like a LinkedIn influencer
    - Never use more than 3-5 hashtags, always at the very end

  HASHTAG RULES:
    - 3-5 hashtags maximum
    - Always at the very end of the post, never inline
    - Mix broad (#AI #DataEngineering) with specific (#NVIDIA #FastAPI)
    - Rotate hashtags — dont use exact same set every post

  EXAMPLES OF GOOD FIRST 2 LINES:
    - "Just tested the new Claude model on our 8000 line codebase. The results
       were not what I expected"
    - "Everyone is talking about GPT-5 but nobody is talking about this"
    - "Built a stock analysis agent that runs 49 modules in parallel. Heres
       what 3 months of building looked like"
    - "Dave Asprey just dropped something interesting about sleep and honestly
       I tried it last week"

7. DUPLICATE PREVENTION + POST HISTORY
==============================================================================

  Database: Separate PostgreSQL database (NOT LuBot's Neon DB)
  Option A: Separate Neon project
  Option B: Local PostgreSQL on Hetzner in Docker

  TABLE: publisher_posts
    id                  SERIAL PRIMARY KEY
    posted_at           TIMESTAMP WITH TIME ZONE NOT NULL
    topic_category      VARCHAR(50) NOT NULL
    topic_title         VARCHAR(500) NOT NULL
    source_url          TEXT
    post_text           TEXT NOT NULL
    image_path          TEXT
    hashtags            TEXT[]
    linkedin_post_urn   VARCHAR(200)
    posting_time_ct     VARCHAR(10)           -- "4:47 PM" for schedule analysis
    day_of_week         VARCHAR(10)           -- "monday" for rotation tracking
    created_at          TIMESTAMP DEFAULT NOW()

  TABLE: publisher_analytics
    id                  SERIAL PRIMARY KEY
    post_id             INTEGER REFERENCES publisher_posts(id)
    linkedin_post_urn   VARCHAR(200) NOT NULL
    checked_at          TIMESTAMP WITH TIME ZONE NOT NULL
    likes               INTEGER DEFAULT 0
    comments            INTEGER DEFAULT 0
    shares              INTEGER DEFAULT 0
    impressions         INTEGER DEFAULT 0
    clicks              INTEGER DEFAULT 0
    engagement_rate     FLOAT DEFAULT 0.0

  TABLE: publisher_topic_performance
    topic_category      VARCHAR(50) PRIMARY KEY
    total_posts         INTEGER DEFAULT 0
    avg_likes           FLOAT DEFAULT 0
    avg_comments        FLOAT DEFAULT 0
    avg_impressions     FLOAT DEFAULT 0
    avg_engagement_rate FLOAT DEFAULT 0
    best_post_id        INTEGER
    worst_post_id       INTEGER
    best_posting_hour   INTEGER               -- best hour for this category
    updated_at          TIMESTAMP DEFAULT NOW()

  TABLE: publisher_scraped_urls
    id                  SERIAL PRIMARY KEY
    url                 TEXT UNIQUE NOT NULL
    scraped_at          TIMESTAMP DEFAULT NOW()
    used                BOOLEAN DEFAULT FALSE

  DUPLICATE PREVENTION LOGIC:
    1. Exact URL check — never post about same article twice
    2. Semantic similarity — NVIDIA embeddings compare new topic against last
       90 days. If cosine similarity > 0.85, skip and pick next candidate
    3. Category balance — query posts per category last 30 days, ensure
       even distribution
    4. Recency — for news categories, never post about news older than 7 days
    5. Title fuzzy match — Levenshtein distance check on topic titles

8. ANALYTICS + SELF-LEARNING ENGINE
==============================================================================

  The system gets smarter over time. After 30 days of data, it knows what
  Lubo's audience wants.

  DAILY ANALYTICS WORKER (analytics_worker.py):
    - Runs every morning
    - Fetches engagement metrics for all posts from last 7 days via LinkedIn API
    - Updates publisher_analytics table
    - Recalculates publisher_topic_performance

  WHAT WE TRACK:
    - Impressions: how many people saw the post
    - Likes: basic engagement
    - Comments: deep engagement (most valuable signal)
    - Shares: viral potential
    - Clicks: are people clicking on screenshots/links
    - Engagement rate: (likes + comments + shares) / impressions
    - Best posting time per category
    - Follower growth: weekly tracking

  SELF-LEARNING FEEDBACK LOOP (self_learner.py):
    - Before writing each post, AI gets a performance report:
      "Your top topics last 30 days:
       1. Biohacker — avg 52 likes, 15 comments (trending UP)
       2. AI News — avg 45 likes, 12 comments
       3. Tech Talk — avg 38 likes, 8 comments
       Worst: Big Tech — avg 12 likes, 2 comments
       Best time: 5:30 PM CT (2x better than 8 PM)"
    - AI adjusts tone, depth, and angle based on feedback
    - Low performers get different angles next time (not dropped entirely)
    - Posting time window adjusts toward best-performing hours

  AFTER 90 DAYS THE SYSTEM KNOWS:
    - Which topics YOUR audience loves
    - What time YOUR followers are most active
    - Whether short or long posts perform better
    - If terminal screenshots outperform website screenshots
    - Which hashtags drive discovery
    - Optimal posting frequency

  GROWTH METRICS DASHBOARD (future):
    - Weekly follower count tracking
    - Month-over-month engagement growth
    - Content performance trends
    - Audience growth rate

9. ANTI-DETECTION + LINKEDIN SAFETY
==============================================================================

  We use the OFFICIAL LinkedIn API (not browser automation), so we are
  already compliant. But extra safety measures:

  POSTING SAFETY:
    - 1 post per day maximum (well within LinkedIn limits)
    - Random posting time (never same time two days in a row)
    - Weekdays 4-10pm CT, weekends 10am-6pm CT
    - Minimum 1 hour difference between consecutive days
    - No burst actions (post + like + comment in same second)

  CONTENT SAFETY:
    - Vary post length (short 400char to long 1500char)
    - Different hashtag sets per post
    - Mix of screenshot types (terminal, website, dashboard)
    - Different hook styles (question, statement, story)
    - No links in post body (uses comment for links — also better reach)

  RATE LIMITS:
    - LinkedIn rate limits reset daily at midnight UTC
    - 1 post/day = nowhere near any limit
    - Monitor via Developer Portal Analytics tab
    - Auto-stop if 429 received (wait 24h, retry)

  TOKEN MANAGEMENT:
    - Access token expires every 60 days
    - token_manager.py checks daily
    - 7 days before expiry: sends notification (email or Slack)
    - Lubo clicks re-auth link (30 seconds, one click)
    - New token saved automatically

10. SECURITY (CRITICAL — same as LuBot, learned from hack)
==============================================================================

  Lubo's server was hacked before. After that, LuBot was rebuilt with
  hardened security. Publisher gets the SAME security from day one.
  No shortcuts. No "its just a personal tool" excuses.

  DISTROLESS CONTAINERS:
    - Google's gcr.io/distroless base images
    - No shell (/bin/sh, /bin/bash removed)
    - No package managers (apt, apk, pip removed)
    - No utility tools (wget, curl, chmod, rm removed)
    - Attack surface: ~0.01% vs ~93% on standard containers
    - Immutable — rebuilt from source on every deploy

  NON-ROOT EXECUTION:
    - All containers run as UID 1000
    - No privilege escalation possible
    - Read-only filesystem inside containers

  SECRETS MANAGEMENT:
    - LinkedIn Client ID/Secret in .env ONLY
    - LinkedIn Access Token in .env ONLY
    - NVIDIA API Key in .env ONLY
    - Database credentials in .env ONLY
    - .env in .gitignore — NEVER committed to git
    - .env in .dockerignore — NEVER baked into images
    - Separate .env per environment (if staging vs prod needed later)

  NETWORK SECURITY:
    - NO public ports exposed (this is a background worker, not a web server)
    - Docker internal network only
    - Outbound connections only: LinkedIn API, NVIDIA API, web scraping
    - No inbound traffic needed at all
    - UFW firewall already configured on Hetzner
    - Tailscale VPN for admin access only
    - More secure than LuBot because zero attack surface from internet

  CODE SECURITY:
    - Ruff linting on every commit
    - Pre-commit hooks prevent committing .env files
    - CI runs full test suite before any deploy
    - No secrets in logs — token values masked in all log output
    - OAuth tokens stored encrypted at rest (Fernet, same as LuBot)

  DOCKER COMPOSE HARDENING:
    - read_only: true (read-only root filesystem)
    - no_new_privileges: true
    - security_opt: no-new-privileges
    - cap_drop: ALL (drop all Linux capabilities)
    - tmpfs: /tmp (writable temp only in memory)
    - Isolated Docker network (not shared with LuBot containers)

  BACKUP SECURITY:
    - Restic backups encrypted with AES-256 to Backblaze B2
    - Same 3-layer backup as LuBot (Forgejo + GitHub mirror + Restic/B2)
    - Database dumps included in nightly backup

  ============================================================================
  - EMBEDDING MODEL: nvidia/llama-nemotron-embed-vl-1b-v2 (NVIDIA NeMo Retriever
    multimodal VLM). 2048-dim, max 8192 tokens. VERIFIED against NVIDIA live docs
    (build.nvidia.com modelcard + docs.api.nvidia.com NIM reference). This is the
    SAME model Lubo shipped in lubot-staging PDF RAG v2 (385 tests) — proven,
    NVIDIA-recommended pattern. Bigger token window than dedup's nv-embedqa-e5-v5
    (1024-dim/512-tok) so our chunks embed with zero truncation.
    - Endpoint: POST https://integrate.api.nvidia.com/v1/embeddings
    - Payload: {input:[...], model, input_type, modality:"text"}
    - input_type="passage" for book chunks, "query" for the search query
    - modality ALWAYS "text" (publisher is text-only — no image embedding)
    - L2-normalize, float32. Model ID via env NVIDIA_VLM_EMBED_MODEL (vendor
      rotations = 1-line .env change). Auth via existing NVIDIA_API_KEY.
    - REPLICATE staging's pattern (tools/nvidia_embeddings.py MultimodalEmbedder),
      do NOT import it — publisher is a separate repo/DB/container.
  - CHUNKING: flat ~400-word chunks, ~50-word overlap, split on paragraph/sentence
    boundaries (never mid-sentence). NOT staging's parent-child — simpler, and
    grounding a post needs less than full document Q&A.
  - STORAGE: PostgreSQL table publisher_knowledge_base (no pgvector). Embedding
    stored as JSON. At query time load all rows once into an in-memory numpy
    matrix (module cache) → one matrix-vector dot product. No FAISS.
  - WHICH POSTS GET BOOK GROUNDING (Lubo, Jun 15): ONLY the technical categories —
    tech_talk, my_agent_git (My Agent Build), ai_news. NOT biohacker, ai_gadgets,
    my_agent, or wakatime (books do not help those).
  - BOOK CITATIONS (Lubo, Jun 15): NEVER name the book in a post. Use the idea in
    Lubo's own words. No "as DDIA says", no titles, no author names. (Avoids
    show-offy tone AND removes any risk of fabricated citations.)
  - RAW PDFs: KEEP them on the server after ingest (gitignored, only 115 MB).
    Lets us re-chunk / re-embed later without re-copying from the Mac.

  ============================================================================
  VOICE & POSITIONING GUARDRAILS (this is what makes or breaks the goal)
  ============================================================================
  These OUTRANK the retrieval logic. A clean authentic post with NO book concept
  beats a post stuffed with a half-relevant one. Order of priority when writing:
    1. Lubos voice (ESL, imperfect grammar, opinionated, truthful) — see voice_rules.yaml
    2. Insight-first: post opens with Lubos REACTION/OPINION, never a fact or a
       book idea (existing rule — book concept must NOT become the hook)
    3. Value to the audience (a senior takeaway they can use)
    4. THEN — and only if it fits — the book concept as invisible background
  - SENIOR NOT STUDENT: the concept informs Lubos OPINION; it is never the subject.
    A senior says "I learned the hard way that X" — the experience leads, the book
    only validates. Never paraphrase the book. Never sound like a textbook.
  - TRUTH GUARD (non-negotiable): book knowledge = things Lubo KNOWS, never things
    he DID. NEVER claim he built/applied/benchmarked something he didn't. If a
    concept can only be used by inventing experience, DROP it. No fake war stories.
  - BETTER NOTHING THAN FORCED: if retrieval is below threshold, inject nothing and
    write the normal authentic post. Silence > generic.
  - NO BOOK NAMES (Lubo's call): never cite/title/author. Removes show-off tone AND
    kills fabricated-citation risk.
  - ai_news is the WEAKEST fit of the 3 grounded categories (news is current, books
    are foundational). Rely on the threshold; if E2E (15c-8) shows forced grounding
    on news, drop ai_news and keep only tech_talk + my_agent_git.

DISTILLATION (the key quality lever - folded in Jun 22 after review)
  Raw ASR is ~10k words with NO structure, plus intro + SPONSOR ad-reads at the
  front and off-topic tangents. Feeding that raw (or naively front-trimmed) to the
  writer is the difference between a sharp post and a mushy one - it would react to
  an ad read or a tangent. So between transcription and the writer we add ONE cheap
  LLM pass that turns the transcript into the writer's actual angle:
  - INPUT: the full cached transcript.
  - OUTPUT: 3-5 bullet points = "the key market arguments / themes / debates the
    hosts made this week" - the meat, in plain words.
  - STRIP: ad-reads, sponsor segments, intros/outros, housekeeping, pure tangents.
  - NO NUMBERS in the bullets (keeps the zero-BS rule airtight downstream).
  - MODEL: latest tech, env-swappable via OPENROUTER_DISTILL_MODEL (a current fast
    capable chat model picked at build time; can reuse the writer model).
  - CACHE: store the distilled bullets alongside the transcript (same guid row), so
    re-runs reuse them too.
  This is what makes the post feel like Lubo genuinely listened and is reacting to
  the real conversation, not skimming show-notes.

GUARDRAILS (same stance as the books/blogs RAG + the zero-BS number check)

==============================================================================
PHASE 2.10c: THEME-TAILORED CHARTS FOR MARKET PULSE  [DONE & LIVE]
==============================================================================

CONTEXT / WHY (Lubo, Jun 22)
  A $300k-caliber post cannot show a generic S&P/Nasdaq/Dow card while the words
  talk about oil, semis, or emerging markets. The chart MUST tell the SAME story as
  the text - text and chart saying different things looks amateur. Until now Market
  Pulse always rendered the 3-index card (Card B) regardless of the post's angle.
  This subphase makes the chart track the post's THEME, staying 100% real data.

KEY INSIGHT (already in place - no renderer change needed)
  The card is already generic: StockInsights(indices={symbol: name}) accepts ANY
  symbols, and build_stock_screenshot_fields renders whatever is in the MarketWeek.
  So we change NOTHING in the chart renderer - we just feed it theme-relevant symbols.

DESIGN - ONE shared dataset drives BOTH text and chart
  Do NOT derive the chart from the post text (fragile) or the post from the chart.
  Pick the theme's symbols ONCE, fetch real yfinance data, and feed the SAME MarketWeek
  to both the writer (numbers) and the chart (series). They then cannot disagree, and
  it is all real data. Flow (theme leads, so the scheduler reorders):
    1. Podcast bullets (Phase 2.10b)
    2. select_chart_symbols(bullets) -> 2-3 real symbols (keyword match, allowlist)
    3. StockInsights(indices=focus).get_market_pulse() -> real data for those
    4. Writer gets that summary + a note: "the chart shows these symbols, build your
       take around them"
    5. Chart renders the SAME MarketWeek
  Fallback: no theme match OR no bullets (podcast failed) -> the 3 indices (today's
  behavior). Always anchor with S&P 500; cap at 3 for a clean card.

SYMBOL SELECTION = deterministic keyword match (NOT the LLM)
  Truthful + deterministic: it can NEVER invent a fake ticker, and the bullets already
  name the themes. A curated allowlist maps a theme keyword -> a real yfinance symbol +
  display name, e.g.:
    broad market/stocks (ANCHOR, always present) -> ^GSPC (S&P 500)
    tech/ai/nasdaq                               -> ^IXIC (Nasdaq) or QQQ
    semis/chips/ai chips/nvidia/semiconductor    -> SMH (Semis ETF)
    oil/crude/energy prices                      -> CL=F (Crude Oil)
    energy sector                                -> XLE
    emerging markets/EM/china/india              -> EEM
    rates/yields/bonds/treasury/fed              -> ^TNX (10Y yield)
    gold                                         -> GC=F
    dollar/dxy                                   -> DX=F
    volatility/vix/fear                          -> ^VIX
    small caps/russell                           -> IWM
  Allowlist is curated + extendable; LLM-assisted selection can come later if wanted.

TRUTH GUARDRAILS (unchanged + reinforced)
  - Only real symbols from the allowlist with real yfinance data are charted/quoted.
  - numbers_grounded() still checks every post number against the SAME (focus-symbol)
    summary -> no fabricated prices/percentages.
  - If a chosen symbol returns no yfinance data, drop it; if none remain, default to
    the 3 indices. Never chart or quote a symbol we do not have real data for.

  C2. scheduler.py market_pulse REORDER + wire: podcast bullets FIRST -> select_chart_
